# DOCUMENTO TÉCNICO INICIAL

## 1. Nombre del Proyecto
**MANTESTOCK: Gestión de inventarios de mantenimiento industrial**

---

## 2. Introducción
Este documento presenta las especificaciones para **ManteStock**, una aplicación pensada para mejorar la gestión de inventarios en entornos de mantenimiento industrial. 

Hoy en día, muchas operaciones de mantenimiento siguen dependiendo de procesos manuales, mensajes dispersos de WhatsApp y hojas de Excel que cada usuario maneja de forma independiente. Esto genera un problema real: **nadie tiene certeza sobre el estado verdadero del inventario**. 

**ManteStock** nace justamente para resolver esto, centralizando la información en un solo sistema confiable que técnicos y coordinadores de compras puedan consultar sin dudas.

---

## 3. Objetivo General
Desarrollar una aplicación de gestión de inventario en tiempo real para el área de mantenimiento, que garantice la precisión de los datos y facilite la toma de decisiones mediante la digitalización de los procesos de despacho y recepción de materiales.

En pocas palabras, ManteStock busca cuatro cosas:
* **Centralizar** la información del inventario en una sola base de datos (dejando atrás Excel y los mensajes informales).
* **Garantizar la integridad** de esos datos evitando saldos negativos e inconsistencias cuando varias personas registran movimientos al mismo tiempo.
* **Dar trazabilidad** a cada movimiento (quién hizo qué y cuándo).
* **Agilizar la operación** dejando que el propio personal de mantenimiento registre sus salidas de repuestos sin depender de intermediarios.

---

## 4. Descripción del Problema
Actualmente, la gestión de inventario en el área de mantenimiento se apoya en herramientas que no fueron diseñadas para este propósito: hojas de cálculo compartidas y mensajería instantánea como WhatsApp. Esto genera una serie de problemas concretos:
* Falta de visibilidad real sobre el stock disponible.
* Compras redundantes de materiales que ya existen en bodega.
* Ausencia de trazabilidad sobre quién retiró qué material y cuándo.
* Inconsistencias en los datos por actualizaciones simultáneas sin control.

---

## 5. Solución del Problema
**ManteStock** centraliza todos los movimientos de inventario en una **única fuente de verdad**, con validaciones automáticas y trazabilidad completa, permitiendo que técnicos y coordinadores de compras trabajen sobre datos confiables y en tiempo real.

### Características Principales
* **Centralización de datos:** Base de datos relacional única para todos los movimientos de inventario.
* **Integridad garantizada:** Validaciones a nivel de backend que previenen saldos negativos e inconsistencias en transacciones simultáneas.
* **Trazabilidad completa:** Cada movimiento queda registrado con responsable y sello de tiempo, facilitando auditorías.
* **Autonomía operativa:** El personal de mantenimiento puede registrar salidas de repuestos de forma autónoma, reduciendo tiempos de espera.

---

## 6. Usuarios Principales
* **Técnico de Mantenimiento:** Consulta la disponibilidad de repuestos en tiempo real, solicita materiales para órdenes de trabajo y registra las salidas de inventario.
* **Coordinador de Compras / Almacén:** Actualiza la entrada de nuevos materiales, gestiona los niveles de stock mínimo y aprueba o rechaza solicitudes de materiales.

---

## 7. Alcance del MVP

### Lo que INCLUYE el MVP (In-Scope)
* Registro de entradas de material a inventario (recepción de compras/reposición).
* Registro de salidas de material (consumo por parte de técnicos de mantenimiento).
* Consulta de stock disponible en tiempo real por parte de técnicos y coordinadores.
* Validación automática que impide registrar salidas superiores al stock disponible (evita saldos negativos).
* Registro de trazabilidad: usuario responsable, fecha/hora y tipo de movimiento (entrada/salida) por cada transacción.
* Gestión básica de usuarios con roles diferenciados (ej. técnico, coordinador de compras).
* Catálogo de materiales/repuestos con identificación única (código, descripción, categoría).

### Lo que queda FUERA del MVP (Out-of-Scope)
* Integración de varios sistemas de software.
* Generación automática de órdenes de compra.
* Reportes avanzados de analítica o predicción de demanda.
* Aplicación móvil nativa (se asume acceso vía navegador web).
* Gestión de proveedores o cotizaciones.

## 8. Requisitos del Sistema

### Requisitos Funcionales (RF)

| ID | Requisito | Descripción |
| :--- | :--- | :--- |
| **RF01** | Registro de material | El sistema debe permitir a coordinador y técnico crear, editar, consultar y eliminar materiales con atributos como código, nombre, categoría y stock actual. |
| **RF02** | Registro de entrada | El sistema debe permitir registrar el ingreso de materiales al inventario, incrementando el stock correspondiente. |
| **RF03** | Registro de salida | El sistema debe permitir registrar la salida y reservas de materiales, decrementando el stock, únicamente si hay disponibilidad suficiente. |
| **RF04** | Validación de stock negativo | El sistema debe rechazar cualquier transacción de salida que resulte en stock negativo, mostrando un código de error claro. |
| **RF05** | Trazabilidad de movimientos | Cada movimiento (entrada/salida) debe quedar asociado al usuario que lo ejecutó y a un timestamp preciso. |
| **RF06** | Consulta de inventario | El sistema debe permitir consultar el stock actual de cualquier material. |
| **RF07** | Historial de movimientos | El sistema debe permitir consultar el historial de movimientos de un material específico (auditoría). |
| **RF08** | Gestión de usuarios y roles | El sistema debe permitir la autenticación de usuarios y diferenciar permisos según su rol (técnico vs. Coordinador). |
| **RF09** | Concurrencia segura | El sistema debe garantizar la consistencia del stock cuando dos o más transacciones ocurren simultáneamente sobre el mismo material. |

### Requisitos No Funcionales (RNF)

| ID | Requisito | Descripción |
| :--- | :--- | :--- |
| **RNF01** | Rendimiento | El sistema debe responder a consultas de stock en menos de 2 segundos bajo condiciones normales de uso. |
| **RNF02** | Seguridad | El acceso al sistema debe requerir autenticación, y las contraseñas deben almacenarse cifradas (hash + salt). |
| **RNF03** | Seguridad | Las acciones críticas (entradas/salidas) deben quedar protegidas por control de permisos según el rol del usuario. |
| **RNF04** | Disponibilidad | El sistema debe estar disponible al menos el 99% del tiempo durante horario operativo (ej. turnos de mantenimiento). |
| **RNF05** | Consistencia de datos | El sistema debe garantizar transacciones atómicas (ACID) al modificar el stock, evitando condiciones de carrera. |
| **RNF06** | Escalabilidad | La arquitectura debe soportar un crecimiento futuro en el volumen de materiales y usuarios sin degradar el rendimiento. |
| **RNF07** | Usabilidad | La interfaz debe ser intuitiva para personal técnico sin experiencia previa en sistemas de gestión, minimizando la curva de aprendizaje. |
| **RNF08** | Auditoría | Los registros de movimientos no deben ser editables ni eliminables una vez creados (integridad del historial). |
| **RNF09** | Compatibilidad | El sistema debe ser accesible desde navegadores web estándar (Chrome, Firefox, Edge) sin requerir instalación adicional. |
| **RNF10** | Mantenibilidad | El código debe seguir una arquitectura modular que facilite la incorporación de nuevas funcionalidades en fases futuras. |

## 9. Módulos del Sistema

### ✓ Módulo de Autenticación y Usuarios
Gestiona el acceso al sistema y los roles de cada usuario.
* Login / logout.
* Registro de usuarios (técnico, coordinador de compras, admin).
* Control de permisos por rol.
* Gestión de sesiones.

### ✓ Módulo de Catálogo de Materiales
Administra la información maestra de los repuestos/materiales.
* Alta, edición y consulta de materiales.
* Categorización (ej. eléctrico, mecánico, consumibles).
* Código único por material.
* Stock actual y stock mínimo (para futuras alertas).

### ✓ Módulo de Inventario / Movimientos
El corazón del sistema: gestiona entradas y salidas.
* Registro de entradas (reposición/compra).
* Registro de salidas (consumo).
* Validación de stock disponible antes de cada salida.
* Control de concurrencia (transacciones atómicas).

### ✓ Módulo de Trazabilidad y Auditoría
Registra y expone el historial de cada movimiento.
* Historial de movimientos por material.
* Historial de movimientos por usuario.
* Filtros por fecha, tipo de movimiento y responsable.
* Registros inmutables (no editables/eliminables).

### ✓ Módulo de Consulta y Dashboard
Interfaz para visualizar el estado del inventario.
* Vista general de stock disponible.
* Búsqueda y filtros de materiales.
* Indicadores básicos (ej. materiales con stock bajo).

### ✓ Módulo de Notificaciones
Alertas simples sobre el estado del inventario.
* Notificación cuando un material llega a stock mínimo.
* Notificación de movimientos críticos.

## 10. Reglas, Límites y Permisos
* El administrador será el encargado de crear, editar o eliminar cuentas o usuarios.
* La interfaz contará con una política de contraseñas (longitud mínima y complejidad) y también la recuperación de esta.
* Contará con bloqueo por intentos fallidos de login (2 intentos, 5 minutos de bloqueo).
* El alcance se realiza para un stock con manejo de decimales y enteros, esto para que no haya variación de materiales.
* Las cantidades de la base de datos de los materiales se manejarán en unidades, cajas o kilogramos
